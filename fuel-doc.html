1. Contexto y propósito

Condugas opera con equipos que consumen gasolina y ACPM (diésel). El combustible se compra en canecas de 55 galones y se despacha en campo, equipo por equipo, persona por persona. Hasta ahora no había forma sistemática de medir rendimientos ni detectar consumos anormales.

Esta aplicación provisional resuelve, sin esperar a Odoo:

  • Registro digital del despacho diario por un operario en celular.
  • Cálculo automático de rendimiento por equipo (hora/galón, km/galón o galones/día según corresponda).
  • Alertas de desviación frente al rendimiento esperado.
  • Tableros semanales/mensuales: top consumidores, equipos críticos, alertas activas.
Para el implementador Odoo: esta app está pensada para vivir hasta que el módulo Odoo esté listo. El modelo de datos ya prevé los puntos de enlace (odoo_*_id en cada entidad) para que la transición sea por reemplazo de fuente, sin perder histórico.

3.5 Cálculo de rendimiento

El rendimiento se calcula entre dos despachos consecutivos del mismo equipo:

MediciónFórmulaUnidadMejor cuando…
horas(horasn − horasn−1) / galonesnhora/galMás alto
odometro(kmn − kmn−1) / galonesnkm/galMás alto
diagalonesn / días entre n y n−1gal/díaMás bajo

Alerta cuando |desviación pct| > alerta_umbral_pct (default 20 %).

4. API REST

Endpoints expuestos en https://<host>/fuel/. Autenticación: HTTP Basic con htpasswd dedicado.

4.1 Equipos

  • GET /fuel/equipos — lista
  • POST /fuel/equipos — alta · body: {codigo, nombre, tipo, combustible, medicion, rendimiento_esperado, alerta_umbral_pct?}
  • PUT /fuel/equipos/<id> — update parcial
  • DELETE /fuel/equipos/<id> — soft-delete (activo=false)

4.2 Personas

  • GET /fuel/personas
  • POST /fuel/personas
  • PUT /fuel/personas/<id>

4.3 Canecas (tanques)

  • GET /fuel/tanques?estado=activo|cerrado|all
  • POST /fuel/tanques — abre nueva
  • PUT /fuel/tanques/<id>/cerrar — cierra
  • PUT /fuel/tanques/<id> — edita notas / código

4.4 Despachos

  • GET /fuel/despachos?desde=&hasta=&equipo=&persona=&tanque=
  • POST /fuel/despachos — devuelve {despacho, rendimiento, galones_restantes_tanque}
  • DELETE /fuel/despachos/<id> — revierte

4.5 Reportes

  • GET /fuel/rendimientos?equipo=&desde=&hasta=
  • GET /fuel/kpis?periodo=semana|mes
  • GET /fuel/alertas
  • GET /fuel/health — liveness

5.3 Canecas → modelo custom condugas.fuel.tank

No hay equivalente nativo en Odoo. Crear modelo nuevo:

class FuelTank(models.Model):
    _name = 'condugas.fuel.tank'
    _description = 'Caneca de combustible'

    name = fields.Char(string='Código', required=True)
    fuel_type = fields.Selection([('gasoline','Gasolina'),('diesel','ACPM')], required=True)
    gallons_initial = fields.Float(string='Galones iniciales', default=55.0)
    gallons_used = fields.Float(compute='_compute_gallons_used', store=True)
    gallons_remaining = fields.Float(compute='_compute_remaining', store=True)
    open_date = fields.Datetime(default=fields.Datetime.now)
    close_date = fields.Datetime()
    state = fields.Selection([('active','Activa'),('closed','Cerrada')], default='active')
    note = fields.Text()
    dispatch_ids = fields.One2many('condugas.fuel.dispatch','tank_id')

5.4 Despachos → modelo custom condugas.fuel.dispatch

class FuelDispatch(models.Model):
    _name = 'condugas.fuel.dispatch'
    _description = 'Despacho de combustible'
    _order = 'dispatch_date desc'

    dispatch_date = fields.Datetime(required=True, default=fields.Datetime.now)
    tank_id        = fields.Many2one('condugas.fuel.tank', required=True, ondelete='restrict')
    vehicle_id     = fields.Many2one('fleet.vehicle', required=True)  # o maintenance.equipment
    operator_id    = fields.Many2one('hr.employee', required=True, string='Operador')
    dispatcher_id  = fields.Many2one('hr.employee', required=True, string='Despachador')
    gallons        = fields.Float(required=True)
    reading        = fields.Float(string='Lectura horómetro/odómetro')
    notes          = fields.Text()
    # Calculados
    yield_value    = fields.Float(compute='_compute_yield', store=True)
    yield_unit     = fields.Char(compute='_compute_yield', store=True)
    deviation_pct  = fields.Float(compute='_compute_yield', store=True)
    alert          = fields.Boolean(compute='_compute_yield', store=True)

El _compute_yield replica la lógica del backend actual: busca el despacho previo del mismo vehicle_id ordenado por dispatch_date y aplica la fórmula según la medición.

6. Estrategia de migración

Fase A — Sincronización dual (recomendada)

Mientras se construye el módulo Odoo, la app sigue siendo la fuente operativa. Se implementa un job (cron o webhook) que:

  1. Lee periódicamente fuel-*.json (o consume /fuel/despachos?desde=).
  2. Crea/actualiza en Odoo los registros faltantes vía XML-RPC.
  3. Actualiza odoo_*_id de vuelta en los JSON para que la app conozca el id Odoo.

Ejemplo de conexión XML-RPC (Python):

import xmlrpc.client
url, db, user, pwd = "https://odoo.condugas.co", "condugas", "api@condugas.co", "******"
common = xmlrpc.client.ServerProxy(f"{url}/xmlrpc/2/common")
uid    = common.authenticate(db, user, pwd, {})
models = xmlrpc.client.ServerProxy(f"{url}/xmlrpc/2/object")

# Crear despacho
new_id = models.execute_kw(db, uid, pwd, 'condugas.fuel.dispatch', 'create', [{
    'dispatch_date': '2026-05-12 14:30:00',
    'tank_id':       42,
    'vehicle_id':    17,
    'operator_id':   88,
    'dispatcher_id': 88,
    'gallons':       5.0,
    'reading':       1006,
}])
Ya existen scripts XML-RPC funcionando en el VPS bajo /root/.openclaw/workspace/scripts/ (test-odoo-connection.py, analisis-gerencial-odoo.py). Pueden usarse como plantilla. Credenciales en /root/.env-odoo.

Fase B — Cambio de fuente

Cuando el módulo Odoo esté validado en producción:

  1. Backup completo de fuel-*.json (snapshot histórico).
  2. Apuntar la UI a la API de Odoo (mismos endpoints, distinto base URL). El backend Python pasa a modo proxy o se reemplaza por las vistas Odoo nativas.
  3. Mantener la URL /fuel/ (al menos en read-only) durante 30 días por si hay reconciliaciones.

Fase C — Decomisionado

Apagar agent-fuel-api.service. Conservar JSON como respaldo frío. Redirigir fuel-*.html al módulo Odoo o eliminar.

7.1 Reglas de validación que aplica el backend

  • Galones > 0 y ≤ saldo de la caneca seleccionada.
  • Tipo de combustible del equipo debe coincidir con el de la caneca.
  • Si equipo.medicion es horas o odometro, la lectura es obligatoria.
  • Tanque debe estar en estado activo.
  • Operador debe existir y estar activo.

8. Seguridad y accesos

  • HTTPS obligatorio (Let's Encrypt).
  • HTTP Basic Auth con htpasswd bcrypt: /etc/nginx/.htpasswd-fuel. Independiente del htpasswd general del VPS.
  • Para agregar usuarios: sudo htpasswd -B /etc/nginx/.htpasswd-fuel <usuario>
  • noindex + X-Robots-Tag en headers para que buscadores no indexen.
  • Rate limit nginx (limit_req zone=write_zone burst=20) sobre escrituras.
  • El despachador queda registrado en cada despacho (campo despachador_id), permitiendo auditoría.
Pendiente para producción: migrar de Basic Auth a un login proper (sesión + cookies) y, si Odoo será proveedor de identidad, usar OAuth2/SSO contra el Odoo de Condugas.

9. Siguiente paso para el implementador

  1. Revisar este documento y validar mapeo a modelos Odoo con la realidad de Condugas (versión de Odoo, módulos instalados, estructura organizacional existente).
  2. Crear módulo condugas_fuel:
    • Definir condugas.fuel.tank y condugas.fuel.dispatch.
    • Agregar campos custom a fleet.vehicle y hr.employee (medición, rendimiento esperado, rol).
    • Vistas: list/form para cada modelo + kanban para canecas activas.
    • Reportes: pivot/graph con galones por equipo, rendimiento promedio, alertas.
  3. Diseñar el endpoint XML-RPC o REST para que la app provisional pueda hacer push/pull durante Fase A.
  4. Plan de carga inicial:
    • Si hay catálogo de vehículos en Odoo, exportar y poblar primero los odoo_equipo_id.
    • Igual con empleados.
    • Canecas y despachos: importar desde JSON una vez los modelos custom existan.
  5. Definir si la app provisional se conservará como UI móvil para campo (más rápida que Odoo) o si se reemplaza por la app móvil de Odoo.
